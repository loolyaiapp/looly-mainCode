Stealth System
How Looly Is Completely Invisible
This is the most critical engineering piece. Looly uses two separate Windows Win32 API calls in Rust to become invisible to screen capture tools AND hidden from the taskbar — all without any third-party library.
1
🪟 App Launches → Rust Gets Window Handle
When Looly starts, the Tauri framework creates a native Windows window (HWND). The Rust backend immediately grabs this handle using window.hwnd() and runs stealth setup before the UI is even visible to the user.
Rust — lib.rs // Tauri setup hook — runs immediately on launch .setup(|app| { let window = app.get_webview_window("main") .expect("main window not found"); // Get the raw Windows HWND (window handle) let hwnd = window.hwnd().expect("failed to get HWND"); // Apply stealth — this is the magic win32::setup_stealth(hwnd.0 as isize); Ok(()) })
Rust Tauri v2 Win32 API HWND
2
🎭 SetWindowDisplayAffinity — Hidden from Screen Capture
This is the core stealth mechanism. Windows has a built-in API used by Netflix, banking apps, and DRM software to prevent screen capture. Looly uses the same API with the value 0x00000011 (WDA_EXCLUDEFROMCAPTURE) to make the window render as a black/empty rectangle in any capture tool.
Rust — Win32 FFI // Direct call to Windows user32.dll #[link(name = "user32")] extern "system" { fn SetWindowDisplayAffinity(hwnd: isize, affinity: u32) -> u32; } pub fn setup_stealth(hwnd: isize) { unsafe { // 0x00000011 = WDA_EXCLUDEFROMCAPTURE // Makes window invisible to: OBS, Win+Shift+S, // PrintScreen, Zoom, Teams, Google Meet SetWindowDisplayAffinity(hwnd, 0x00000011); } }
🔬
Why this works technically
Windows Desktop Window Manager (DWM) handles compositing. When WDA_EXCLUDEFROMCAPTURE is set, DWM tells the capture API to replace the window's pixels with black/transparent in the captured frame — while still rendering it normally on the physical display. The GPU renders two versions: one for the monitor, one for capture APIs. Looly only appears in the monitor output.
3
📌 WS_EX_TOOLWINDOW — Hidden from Taskbar & Alt+Tab
Even though the window is invisible to captures, it would normally appear in the Windows taskbar and Alt+Tab switcher — potentially alerting the interviewer. A second Win32 call modifies the extended window style to set it as a "tool window", which Windows automatically excludes from the taskbar, Alt+Tab, and Task View.
Rust — Win32 FFI extern "system" { fn GetWindowLongW(hwnd: isize, index: i32) -> i32; fn SetWindowLongW(hwnd: isize, index: i32, val: i32) -> i32; } const GWL_EXSTYLE: i32 = -20; const WS_EX_TOOLWINDOW: i32 = 0x00000080; // Hide from taskbar const WS_EX_APPWINDOW: i32 = 0x00040000; // Remove app window flag unsafe { let style = GetWindowLongW(hwnd, GWL_EXSTYLE); let new_style = (style | WS_EX_TOOLWINDOW) & !WS_EX_APPWINDOW; SetWindowLongW(hwnd, GWL_EXSTYLE, new_style); }
4
⌨️ Global Hotkey — Show/Hide Even When Minimized
Since Looly has no taskbar icon, users need another way to bring it back. A system-wide global shortcut Ctrl+Shift+Space is registered at the OS level — it works even when the window is hidden, minimized, or behind other apps.
Rust — Global Shortcut app.global_shortcut().register( Shortcut::new( Some(Modifiers::CONTROL | Modifiers::SHIFT), Code::Space ) )?; // Handler: toggle show/hide if minimized || !visible { window.show().ok(); window.unminimize().ok(); window.set_focus().ok(); } else { window.minimize().ok(); }

Input Modes
3 Ways to Capture Questions
Looly has three distinct input modes. Each feeds into the same AI engine but captures the interview question differently.
A
🎤 Listen Mode — Real-Time Speech Recognition
The most powerful mode. Looly uses the browser's built-in Web Speech API to continuously transcribe everything being said in the interview. When the interviewer stops speaking for 4 seconds, Looly auto-sends the question to AI.
User clicks 🎤 │ ▼ SpeechRecognition starts (continuous=true, interimResults=true) │ ├─ Interviewer speaks ──────► Interim text shown faded in textarea │ (updates in real-time as they speak) │ └─ Word finalized ──────────► Final transcript appended (locked in, shown solid) │ ▼ 4-second silence detected │ ├─ Countdown: 4... 3... 2... 1... shown in UI │ ├─ User speaks again? ──► Reset countdown to 4s │ └─ Countdown reaches 0 ──► Auto-send to AI
⚡
Smart Queue System
If a new question comes in while AI is still answering, Looly queues it instead of dropping it. A "next queued" badge appears in the UI, and the pending question runs automatically the moment the current answer finishes streaming.
B
⌨️ Type Mode — Manual Question Entry
For questions shown on screen (LeetCode, HackerRank, coding platforms). The user pastes or types the problem directly into the textarea. Press Ctrl+Enter or click Ask to send it instantly.
📋
Paste question from coding platform
User copies the problem statement from LeetCode/HackerRank/CodeSignal and pastes into Looly's textarea. Works with any text.
⌨️
Press Ctrl+Enter to ask
Global keyboard shortcut works anywhere in the app. No mouse needed — keeps hands on keyboard during coding interviews.
🧹
Ctrl+L to clear instantly
After getting the answer, one shortcut clears everything and aborts any in-flight AI stream. Ready for the next question immediately.
C
📷 Screenshot Mode — Visual Question Analysis
For questions shown visually on screen — diagrams, whiteboard drawings, system design images. A single click captures the entire screen, compresses it, and sends it to a vision AI model that can see and understand images.
User clicks 📷 │ ▼ Rust command invoked: invoke("take_screenshot") │ ▼ screenshots crate captures full screen └─ Returns RgbaImage (raw pixel data) │ ▼ Rust encodes PNG → base64 string └─ Sent back to React frontend │ ▼ Frontend: Canvas API resizes to 1280px width └─ Re-encodes as JPEG at 75% quality └─ Reduces from ~8MB PNG → ~200-400KB JPEG │ ▼ Thumbnail preview shown in UI (with ✕ to discard) │ ▼ User clicks "Analyze Screenshot" │ ▼ Sent to LLaMA 4 Scout Vision Model └─ model: "meta-llama/llama-4-scout-17b-16e-instruct" └─ message format: [image_url + text prompt]

AI Engine
How the AI Generates Answers
Looly has two AI paths depending on whether the user is on the Free plan (their own Groq key) or Pro (backend key pool). Both stream answers token by token for instant feedback.
🆓 FREE PATH
  • User provides their own Groq API key
  • Groq SDK called directly from frontend
  • No backend involved for AI requests
  • Model: llama-3.3-70b-versatile (default)
  • User-selectable from 4 available models
  • Rate limits apply to user's own key
  • 10-day free trial period
⭐ PRO PATH
  • No API key needed from the user
  • All AI calls routed through backend
  • Backend has a pool of 6 Groq API keys
  • Round-robin rotation across keys
  • 429 rate limit → automatically try next key
  • Vision model for screenshots: LLaMA 4 Scout
  • Unlimited questions, no rate limit friction
1
🧠 Question Reaches the AI
The question (text or image) is packaged into a Groq API request. The system prompt defines Looly's personality: elite interview assistant, code-first answers, STAR format for behavioral questions, direct and confident.
JavaScript — Backend ask.js const messages = imageBase64 ? [{ role: "user", content: [ { type: "image_url", image_url: { url: `data:image/jpeg;base64,${imageBase64}` }}, { type: "text", text: questionText } ]}] : [ { role: "system", content: SYSTEM_PROMPT }, { role: "user", content: question } ]; const stream = await groq.chat.completions.create({ model: useModel, // llama-3.3-70b OR llama-4-scout (vision) stream: true, // token-by-token streaming max_tokens: 1500, messages, });
2
📡 Tokens Stream via Server-Sent Events (SSE)
For Pro users, the backend doesn't wait for the full answer before responding. It opens an SSE connection and pipes each token from Groq directly to the frontend as it arrives. This gives the "typing in real time" effect.
JavaScript — SSE streaming // Backend: set SSE headers res.setHeader("Content-Type", "text/event-stream"); res.setHeader("Cache-Control", "no-cache"); res.flushHeaders(); // For each chunk from Groq: for await (const chunk of stream) { const text = chunk.choices[0]?.delta?.content || ""; if (text) res.write(`data: ${JSON.stringify({ text })}\n\n`); } res.write("data: [DONE]\n\n"); res.end();
3
🖥️ Frontend Renders Answer Token by Token
The frontend reads the SSE stream, parses each data: {...} line, and appends tokens to the response. Code blocks are detected with regex and rendered in a separate styled block. The cursor blink animation makes it feel live.
TypeScript — Frontend stream reader const reader = res.body!.getReader(); const decoder = new TextDecoder(); let buffer = ""; while (true) { const { done, value } = await reader.read(); if (done) break; buffer += decoder.decode(value, { stream: true }); const lines = buffer.split("\n"); buffer = lines.pop()!; for (const line of lines) { if (!line.startsWith("data: ")) continue; const { text } = JSON.parse(line.slice(6)); if (text) setResponse(prev => prev + text); // append each token } }
🎨
Smart Code Block Parsing
After streaming, a regex parser scans the full response for triple-backtick code blocks. Code sections get a dark background, monospace font, language label, and overflow scroll. Regular text gets normal line spacing. This happens live as tokens arrive.
Payment & Licensing
From Payment to Active License
The full lifecycle from a user clicking "Buy" to having an active Pro license in the desktop app — including payment verification, key generation, email delivery, and in-app activation.
1
🔐 Sign In Required → Google OAuth
When a user clicks "Get Monthly", "Get Annual", or "Download Free", they are automatically redirected to Google Sign-In if not already authenticated. After sign-in, they are returned to the page and their action resumes automatically.
User clicks "Get Monthly" │ ▼ requireAuthThen('monthly') called │ ├─ Already signed in? ──────► Open payment modal immediately │ └─ Not signed in? │ ▼ localStorage.setItem("looly_pending_action", "monthly") │ ▼ Redirect → /api/auth/google │ ▼ Supabase → Google OAuth (user logs in) │ ▼ Redirect back → landing page #access_token=... │ ▼ Token saved, user loaded, pending action executed
2
💳 Razorpay Payment (UPI / Cards / Netbanking)
Frontend creates a Razorpay order via the backend, then opens the Razorpay modal. Supports UPI (GPay, PhonePe, Paytm), debit/credit cards, and netbanking — all in INR with no international card required.
JavaScript — Payment Flow // Step 1: Create order on backend const order = await fetch("/api/razorpay/create-order", { method: "POST", body: JSON.stringify({ amount: 470000, plan: "pro-monthly", email }) }); // Step 2: Open Razorpay modal const rzp = new Razorpay({ key_id: order.keyId, amount: order.amount, currency: "INR", handler: async (response) => { // Step 3: Verify payment signature on backend await verifyPayment(response, email, plan); } });
3
✅ Payment Verified → License Generated
The backend verifies the Razorpay HMAC signature to confirm the payment is genuine (not tampered). Then generates a unique license key in LOOLY-XXXX-XXXX-XXXX-XXXX format and stores it in Supabase.
JavaScript — Signature Verification // Verify Razorpay HMAC signature const sign = `${razorpay_order_id}|${razorpay_payment_id}`; const expected = crypto .createHmac("sha256", process.env.RAZORPAY_KEY_SECRET) .update(sign) .digest("hex"); if (expected !== razorpay_signature) return res.status(400).json({ error: "Invalid signature" }); // Generate license key: LOOLY-XXXX-XXXX-XXXX-XXXX const license = await createLicense({ email, plan, paymentId }); // Email sent async (doesn't block response) sendLicenseEmail({ to: email, key: license.key, plan }) .catch(e => console.error(e)); // Return key immediately res.json({ licenseKey: license.key });
4
🔑 License Activated in Desktop App
User pastes the LOOLY-XXXX key into the app settings. The app calls the backend to validate it, stores a 24-hour cache so it works offline, and unlocks Pro mode — routing all AI calls through the backend key pool.
User pastes LOOLY-XXXX key │ ▼ Check 24hr local cache ├─ Cache hit + valid? ──────────► Activate Pro (offline capable) │ └─ No cache / expired │ ▼ tauriFetch → POST /api/verify-license (Tauri HTTP plugin — bypasses WebView2) │ ▼ Backend checks Supabase licenses table │ ┌───────┴────────┐ valid expired/invalid │ │ ▼ ▼ Save to cache Show error message Set Pro mode Stay in free mode
🔧
Critical Fix: Tauri HTTP Plugin Scoping
Standard fetch() in a Tauri WebView2 window is blocked by Windows Firewall and WebView2 network restrictions in production builds. The solution was to use @tauri-apps/plugin-http (native Rust HTTP) AND explicitly declare allowed URLs in the capabilities manifest. Without this, all API calls silently fail.
